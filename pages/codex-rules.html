CodexにおけるRulesとは？

コマンドを自動許可・毎回確認・禁止するための実行ポリシー

記事IDcodex-rules

結論

CodexのRulesは、Codexがサンドボックス外で特定のコマンドを実行しようとした時に、自動許可するか、確認を求めるか、禁止するかを決めるルールです。

たとえば、git commitは自動許可し、公開につながるgit push origin mainは毎回確認する、といった制御ができます。文章の書き方や開発方針を教えるものではなく、主にコマンド実行の境界を管理する仕組みです。

Rulesが必要な理由

Codexはファイル編集やコマンド実行を行えますが、すべてを無条件に許可すると、誤操作や意図しない公開につながります。一方、毎回すべてを確認すると作業が止まりがちです。

Codexコマンドを提案
→
Rules許可方針を判定
→
実行結果許可・確認・禁止

Rulesは、よく使う安全な操作だけを自動化し、重要な操作には人間の判断を残すための中間層です。

3つの判定

decision動作適した例
allowサンドボックス外で確認なしに実行を許可限定したgit add、テスト実行
prompt実行前にユーザーへ確認git push、外部公開、ネットワーク操作
forbidden確認せずブロック組織で禁止する危険なコマンド

複数のルールが一致した場合は、公式仕様上forbidden → prompt → allowの順で、より厳しい判定が優先されます。

基本構造：prefix_rule

prefix_rule(
    pattern = ["git", "commit"],
    decision = "allow",
    justification = "Allow creating local commits",
)
項目意味
pattern照合するコマンドの先頭部分。必須です。
decisionallow、prompt、forbiddenから選びます。省略時はallowです。
justificationその判定にした理由。承認画面などに表示される場合があります。
match / not_match想定したコマンドが一致・不一致になるか検証する例です。

patternはコマンド文字列全体ではなく、引数を分けた配列として指定します。

Git操作を分けて考える例

prefix_rule(
    pattern = ["git", "add"],
    decision = "allow",
)

prefix_rule(
    pattern = ["git", "commit"],
    decision = "allow",
)

prefix_rule(
    pattern = ["git", "push", "origin", "main"],
    decision = "prompt",
)

この例では、手元で履歴を作るまでは自動化し、publicリポジトリへ送信する直前だけ確認します。pattern = ["git"]のように広く許可すると、意図しないGit操作まで対象になるため、必要なサブコマンドまで絞るのが基本です。

どこに保存される？

場所範囲注意
~/.codex/rules/default.rulesそのユーザーのCodex全体個人用。通常はリポジトリに公開されません。
.codex/rules/*.rules信頼済みプロジェクトプロジェクト設定として共有され得ます。
Team Configのrules組織・チーム管理者が共通方針を配布できます。

Codexは起動時に有効な設定階層のrules/を読み込みます。ファイルを追加・変更した後はCodexを再起動するのが確実です。承認画面で永続的な許可を選んだ場合も、ユーザー側のrulesへ保存されます。

AGENTS.mdやskillsとの違い

仕組み決めるもの例
Rulesコマンドを実行してよいかgit push前に確認する
AGENTS.mdリポジトリ内で守る開発方針テスト方法、命名規則、編集範囲
Skill特定タスクの再利用可能な手順記事作成、PDF生成、レビュー手順
config.tomlCodexの全体的な動作設定モデル、承認方針、MCP設定

「pushしてよいか」はRules、「質問！なら記事を作る」はSkill、「このリポジトリではこのテストを実行する」はAGENTS.md、という分担です。

サンドボックスとの関係

Rulesとサンドボックスは同じものではありません。

  • サンドボックス：通常の実行で、技術的にどこまで読み書き・通信できるかを制限します。
  • Rules：サンドボックス外での実行が必要な時、そのコマンドをどう扱うか判定します。
  • 承認ポリシー：どの場面で人間や自動レビュアーへ確認するかを決めます。

Rulesでallowにしても、OS権限や組織の管理ポリシーを無効化できるわけではありません。より上位の制限があれば、そちらが優先されます。

複合コマンドはどう判定される？

たとえば次の操作を考えます。

git add . && rm -rf /

安全に解析できる単純な連結なら、Codexは各コマンドへ分割してルールを評価します。git addを許可していても、後半の危険な操作まで自動許可されるわけではありません。

ただし、変数展開、リダイレクト、ワイルドカード、複雑な制御構文が入る場合は保守的に扱われます。ルールを安定して効かせるには、一つの実行に複雑なシェル処理を詰め込まないことが大切です。

設定をテストする

codex execpolicy check --pretty \
  --rules ~/.codex/rules/default.rules \
  -- git push origin main

このコマンドは、実際にpushせず、どのルールが一致し、最終的にどの判定になるかを表示します。ルールを追加した時は、許可したい例だけでなく、許可したくない似たコマンドも確認すると安全です。

つまり何が大事？

Rulesは、Codexを便利にするだけでなく、自動化してよい境界を明文化する安全装置です。よく使う低リスク操作は狭いprefixで許可し、公開・削除・外部送信などの操作は確認を残すのが実用的です。

なお、Codex公式ではRulesは実験的機能とされており、仕様が変わる可能性があります。設定後は再起動し、codex execpolicy checkで実際の判定を確認してください。

理解度チェック

  1. Q1. CodexのRulesが主に制御するものはどれ？

    1. 生成文章の文体だけ
    2. サンドボックス外で実行するコマンドの許可方針
    3. LLMの学習データ
    4. GitHubリポジトリの公開・非公開設定そのもの
    答えを見る

    正解はB。Rulesはコマンドをallow、prompt、forbiddenのどれで扱うか決めます。文章方針はAGENTS.mdやSkillなど別の仕組みが担当します。

  2. Q2. git commitは自動化し、git push origin mainは確認したい場合、適切なのはどれ？

    1. pattern=["git"]をallowにする
    2. 両方をforbiddenにする
    3. commitをallow、push origin mainをpromptに分ける
    4. AGENTS.mdへ「全部許可」とだけ書く
    答えを見る

    正解はC。リスクの違う操作を別々の狭いprefixで管理します。AではGitの広い範囲が許可され、Dは実行権限の設定になりません。

  3. Q3. allowルールがあるのにコマンドが禁止された場合、あり得る理由はどれ？

    1. Rulesでは複数の判定を扱えないから
    2. より厳しいprompt・forbiddenルールや管理ポリシーが一致したから
    3. allowは必ず一度しか動かないから
    4. RulesはGit以外には使えないから
    答えを見る

    正解はB。複数ルールでは最も厳しい判定が優先され、組織管理の制約もユーザー設定より上位になり得ます。
