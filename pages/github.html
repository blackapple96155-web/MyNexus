ひとことで言うと

GitHub は、プログラムのコードや文章などのファイルをネット上に置き、 変更履歴を残しながら、複数人で安全に編集・共有するためのサービスです。 ざっくり言えば、開発者向けのクラウド保管庫であり、 「誰が、いつ、何を変えたか」が見える共同作業の場所です。

まず Git と GitHub は別物

名前何をするもの？
Gitファイルの変更履歴を記録する道具。手元のPCでも使える。
GitHubGit の履歴をネット上で共有し、レビューや公開もできるサービス。
たとえ話：Git は「編集履歴を残せるノート術」、GitHub は「そのノートをチームで置いておくオンライン本棚」みたいなものです。

GitHubでできること

  • コードを保管する：プロジェクトごとにリポジトリを作り、ファイル一式を管理する。
  • 変更履歴を見る：いつ、誰が、どの行を変えたかを追える。
  • 共同編集する：複数人が別々に作業し、あとで変更を合流できる。
  • レビューする：Pull Request で変更内容を確認し、コメントしてから取り込める。
  • Webページを公開する：GitHub Pages を使うと、HTML などを公開サイトにできる。

基本用語の地図

用語意味
リポジトリプロジェクトのフォルダ一式。コード、画像、HTML、履歴などをまとめて置く場所。
コミット変更のセーブポイント。「何を変えたか」をメッセージ付きで記録する。
ブランチ本線から分かれた作業場所。試作や修正を本線に直接混ぜずに進められる。
Pull Request「この変更を本線に入れてください」という提案。レビューの場にもなる。
push手元の変更を GitHub に送ること。
pullGitHub 側の変更を手元に取り込むこと。

MyNexus もこの流れで動いています。HTML ページを作り、`data/pages.json` を更新し、コミットして GitHub に push すると、GitHub Pages で公開されます。

公開・非公開・権限の考え方

GitHub のリポジトリには、誰でも見られる public と、権限を持つ人だけが見られる private があります。 public は作品やドキュメントの公開に便利ですが、置いた内容は世界中から見える前提で扱う必要があります。

状態向いている用途注意点
public公開サイト、OSS、学習メモ、ポートフォリオ秘密情報や個人情報を絶対に入れない
private未公開プロジェクト、個人メモ、仕事用のコード共有相手の権限設定を確認する

Pull Request は「変更の相談部屋」

Pull Request は、単なるアップロードではありません。 「この変更を本線に入れてよいですか？」と提案し、差分を見ながらレビューできる場所です。 チーム開発では、いきなり本線へ混ぜずに Pull Request を通すことで、ミスや意図のずれを見つけやすくなります。

  1. ブランチを作って変更する。
  2. コミットして GitHub に push する。
  3. Pull Request を作り、差分・目的・確認点を書く。
  4. レビューを受け、必要なら修正する。
  5. 問題なければ main などの本線へ merge する。

GitHub Pages と Actions

GitHub はコード置き場にとどまりません。たとえば GitHub Pages を使うと、HTML やCSSで作ったサイトを公開できます。 また GitHub Actions を使うと、push したタイミングでテストやビルド、デプロイを自動実行できます。

誤解しやすいポイント

  • GitHub = Git ではない：Git は履歴管理の道具、GitHub はそれを共有・レビュー・公開するサービスです。
  • 削除すれば完全に消えるとは限らない：Git は履歴を大事にするため、過去コミットに残る場合があります。
  • push は公開と同じとは限らない：private リポジトリへの push は限定共有ですが、public なら世界に見える前提です。
  • main は大事な本線：壊れた変更を入れると公開サイトやチーム全体に影響することがあります。

公開リポジトリで注意すること

  1. 公開範囲を意識する：public リポジトリは誰でも見られます。
  2. 秘密を書かない：APIキー、トークン、パスワード、個人情報は絶対に置かない。
  3. 履歴にも注意する：一度コミットした内容は、削除しても履歴に残ることがあります。
  4. 公開前に確認する：push する前に、公開してよいファイルだけが対象か確認する。

何がうれしいの？

GitHub の強みは、単にファイルを置けることではなく、変更の理由と流れが残ることです。 うまくいった版に戻したり、誰かの変更をレビューしたり、Webサイトを自動公開したりできます。 個人のメモから大きなソフトウェア開発まで、同じ考え方で管理できるのが便利なところです。

つまり GitHub は「保存場所」ではなく、変更を説明し、確認し、必要なら公開までつなげる作業基盤です。 便利さと同時に、公開範囲と履歴の扱いに気を配ることが大切です。

理解度チェック

  1. Q1. Git と GitHub の関係として最も正しいものはどれ？

    1. GitHub は Git の別名で、完全に同じもの
    2. Git は履歴管理の道具で、GitHub はその履歴を共有・レビュー・公開するサービス
    3. Git はWebサイト公開専用で、GitHub は画像編集専用
    4. GitHub を使うと Git の履歴は不要になる
    答えを見る

    正解は B。Git は手元でも使える履歴管理の仕組みで、GitHub はそれをネット上で共有し、Pull Request や Pages などにつなげるサービスです。

  2. Q2. Pull Request を使う主な理由として最も適切なのはどれ？

    1. 変更を本線へ入れる前に、目的・差分・影響を確認しやすくするため
    2. commit せずに変更を永久保存するため
    3. public リポジトリを自動的に private にするため
    4. GitHub Pages を完全に無効化するため
    答えを見る

    正解は A。Pull Request は「変更の相談部屋」で、本線に混ぜる前にレビューや修正をしやすくします。

  3. Q3. publicリポジトリで秘密情報を一度commitしてしまった場合の理解として最も適切なのはどれ？

    1. ファイルを削除すれば履歴からも必ず完全に消える
    2. 履歴に残る可能性があるため、キーの無効化や履歴削除など追加対応が必要になる
    3. GitHub Pages を使っていなければ誰にも見えない
    4. commit メッセージを変えれば秘密情報も自動で伏せ字になる
    答えを見る

    正解は B。Git は履歴を残すため、削除コミットだけでは過去履歴に残る場合があります。漏れたキーは無効化し、必要なら履歴を書き換える対応が必要です。
