MCPサーバーとは？

AIと外部のデータ・道具を、共通の方式でつなぐ窓口

結論

MCPサーバーは、AIアプリに対して「利用できる道具やデータはこれです」と、標準化された形で提供するプログラムです。MCPは Model Context Protocol の略で、AIアプリとファイル、データベース、Webサービスなどをつなぐ共通規格です。

たとえばGitHub用MCPサーバーを接続すると、対応するAIアプリは、そのサーバーが許可した範囲でリポジトリの情報取得やIssue操作などを行えます。サーバーがAIになるのではなく、AIが外部機能を使うための接続口になります。

まず「MCP」を分解する

語意味ここでの役割
Model言語モデルを利用するAIアプリ情報を判断し、必要な機能を選ぶ
Context回答や作業に必要な外部情報ファイル、検索結果、DBの内容など
Protocol通信の共通ルール異なるアプリとサービスの会話方法をそろえる

MCP自体は特定のAIモデルではなく、異なる製品間で接続方法をそろえるためのオープンなプロトコルです。

登場人物は3つ

MCPホストAIアプリ全体
→
MCPクライアント接続を管理
→
MCPサーバー機能や情報を提供
  • ホスト：ユーザーが操作するAIアプリ。会話や権限確認、モデルとの連携をまとめます。
  • クライアント：ホスト内部で、特定のMCPサーバーとの接続を維持します。
  • サーバー：外部サービスやローカル環境を、MCPの形式で公開します。

日常会話ではホストとクライアントをまとめて「MCPクライアント」と呼ぶこともありますが、仕様上は役割が分かれています。

MCPサーバーが提供するもの

機能役割例
ToolsAIが呼び出せる操作検索、Issue作成、SQL実行、ファイル更新
Resources読み取って文脈にできるデータファイル内容、スキーマ、ログ、文書
Prompts再利用できる作業テンプレートレビュー手順、定型調査、要約のひな型

すべてのサーバーが3種類を全部提供する必要はありません。検索だけを提供するサーバーや、社内文書をリソースとして公開するサーバーも作れます。

実際にはどう動く？

  1. AIアプリがMCPサーバーへ接続し、対応するプロトコル版や機能を確認します。
  2. サーバーが利用可能なtools、resources、promptsなどを通知します。
  3. ユーザーの依頼を受けたモデルが、必要なツールや情報を判断します。
  4. AIアプリがMCP経由でサーバーへ要求を送り、サーバーが外部処理を実行します。
  5. 結果がAIアプリへ戻り、モデルがその結果を使って回答や次の処理を組み立てます。

重要なのは、モデルが直接データベースやGitHubへ接続するのではなく、通常はAIアプリとMCPサーバーが実行・通信を管理することです。

ローカル型とリモート型

方式置かれる場所向いている用途主な注意
ローカルMCPサーバー自分のPC上ローカルファイル、開発ツール、個人環境インストールしたコードにPC上の権限を与える
リモートMCPサーバーネットワーク上SaaS、共有DB、チーム向けサービス認証、通信先、送信データの管理が必要

ローカルでは標準入出力を使う方式、リモートではHTTPを使う方式が代表的です。どちらでも、MCPは機能の説明や要求・応答の形をそろえます。

APIやプラグインとの違い

概念主眼MCPとの関係
APIサービス固有の機能をプログラムへ公開するMCPサーバーが背後でAPIを呼ぶことがある
プラグイン特定アプリへ機能を追加する配布単位内部にMCPサーバーを含む場合がある
MCPAIアプリ向けの接続方法を共通化する複数のAIアプリから同じ形式で扱いやすくする

MCPはAPIを消すものではありません。サービスごとに異なるAPIやローカル処理を、AIアプリ側から共通の方法で見えるようにするアダプターの規格と考えると理解しやすくなります。

便利さとセキュリティは表裏一体

  • 信頼できる配布元を選ぶ：ローカルサーバーは、通常のアプリをインストールするのと同じ慎重さで確認します。
  • 最小権限にする：必要なフォルダ、リポジトリ、API権限だけを許可します。
  • 書き込み操作を確認する：削除、送信、購入、公開などは人間の承認を挟む設計が重要です。
  • 秘密情報を渡しすぎない：APIキーは公開設定や会話本文に書かず、安全な秘密管理を使います。
  • 説明だけを信用しない：ツール名が安全そうでも、実装コードと実際のアクセス範囲を確認します。

公式の信頼モデルでも、クライアントは接続したサーバーを信頼し、ローカルサーバーはインストールしたソフトウェア同様のアクセスを持ち得るとされています。MCPにつないだから自動的に安全になるわけではありません。

具体例で整理する

「このリポジトリで失敗中のIssueを調べ、修正案をまとめて」という依頼を考えます。

  1. GitHub用MCPサーバーがIssue検索ツールを公開します。
  2. AIアプリがツールの説明と入力形式をモデルへ知らせます。
  3. モデルが検索条件を決め、AIアプリがツール呼び出しを実行します。
  4. サーバーがGitHub APIへ問い合わせ、結果を返します。
  5. モデルがIssueの内容を読み、修正案をまとめます。

この例では、GitHub APIが実際のサービス窓口、MCPサーバーがAI向けの翻訳・制御層、AIアプリが判断と実行をまとめる側です。

つまり何が大事？

MCPサーバーは、AIに知識を学習させ直す仕組みではありません。必要な時に外部情報を読み、許可された操作を実行できるようにする仕組みです。

MCP＝共通の接続ルール、MCPサーバー＝道具とデータを提供する側、AIアプリ＝接続と実行を管理する側と覚えると、全体像を外しにくくなります。そして導入時は、機能の多さよりも「何へアクセスでき、どの操作を実行できるか」を確認することが重要です。

理解度チェック

  1. Q1. MCPサーバーの説明として最も適切なのはどれ？

    1. LLMそのものを追加学習する専用GPU
    2. AIアプリへ外部のツールやデータを標準形式で提供するプログラム
    3. あらゆるAPIを不要にする単一のクラウドサービス
    4. AIの回答を保存するだけの文章ファイル
    答えを見る

    正解はB。MCPサーバーはモデル本体ではなく、外部機能をMCPの形式で公開する接続口です。背後で既存APIを利用することもあるため、Cでもありません。

  2. Q2. AIがMCPツールを使う時の流れとして適切なのはどれ？

    1. モデルが必ず外部サービスへ直接ログインする
    2. サーバーがモデルを再学習してから結果を返す
    3. サーバーが機能を提示し、AIアプリが要求を送り、実行結果をモデルの文脈へ戻す
    4. ユーザーの許可に関係なくサーバーが全ファイルを公開する
    答えを見る

    正解はC。機能の発見、要求、外部処理、結果の返却という流れです。モデルが外部サービスへ直接接続するとは限らず、公開範囲もサーバー設定と権限で決まります。

  3. Q3. 未知のローカルMCPサーバーを導入する際、最も妥当な判断はどれ？

    1. MCP対応なら安全性は保証されるので、そのまま全権限を与える
    2. ツール名だけ確認し、実装や配布元は確認しない
    3. 配布元、コード、必要権限、アクセス範囲を確認し、最小権限で実行する
    4. ローカル型ならネットワークやファイルへ絶対にアクセスできないと考える
    答えを見る

    正解はC。MCPは通信規格であって安全性の保証ではありません。ローカルサーバーも通常のソフトウェアと同様に、与えられた環境の権限を利用し得ます。
